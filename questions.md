# Вопросы для самопроверки

## 1. Какие критерии вы использовали при разделении на микросервисы?

Система разделена по принципу **разделения ответственности (SRP)** и **разделения протоколов**.

`gateway` отвечает исключительно за взаимодействие с внешним миром: принимает REST-запросы, валидирует входные данные через Pydantic, транслирует в gRPC. Он не знает, как хранятся лайки.

`likes-svc-s03` отвечает только за бизнес-логику: хранить, создавать, отдавать и удалять лайки. Он не знает, кто его вызывает — REST-клиент или другой микросервис.

Такое разделение позволяет масштабировать и обновлять сервисы независимо. Если завтра понадобится добавить GraphQL-интерфейс, нужно будет добавить только ещё один gateway, не трогая likes-svc.

## 2. Почему выбран именно этот стек?

**Python** — язык курса, есть отличные библиотеки для обоих протоколов.

**FastAPI** — минимальный бойлерплейт, автоматическая валидация через Pydantic, встроенный OpenAPI. Для учебного проекта это оптимально.

**gRPC + protobuf** — строгий типизированный контракт. Если изменить `.proto` так, что клиент и сервер несовместимы, это сразу заметно при сборке, а не в рантайме. Для межсервисного общения это важнее, чем удобство curl-отладки.

**In-memory хранилище** — намеренное упрощение. В реальной системе здесь был бы Redis или PostgreSQL. Для финального проекта добавление БД усложнило бы инфраструктуру без учебной ценности.

**Docker + Docker Compose** — стандарт де-факто для локальной разработки. Один `docker compose up` — и среда воспроизведена у любого разработчика.

## 3. Как сервисы справляются с ошибками?

**gRPC-уровень (likes-svc):**
- `NOT_FOUND` (код 5) — возвращается при обращении к несуществующему ID. Клиент получает структурированную ошибку, а не падение.
- `context.abort()` прерывает стрим корректно, без исключений в логах.

**REST-уровень (gateway):**
- Все gRPC-ошибки перехватываются в `except grpc.RpcError` и транслируются в HTTP-коды: `NOT_FOUND → 404`, все остальные → `502 Bad Gateway`.
- Если `likes-svc` недоступен (сеть, перезапуск), клиент получает `502`, а не timeout.

**Что отсутствует (и что добавил бы в реальном проекте):**
- **Retry** — повторные попытки при временных сетевых ошибках (grpc `wait_for_ready=True` или tenacity).
- **Circuit breaker** — после N ошибок подряд прекратить попытки и сразу отдавать 503, чтобы не накапливать очередь.
- **Timeout** — сейчас gRPC-вызовы без явного таймаута; нужно добавить `timeout=5` в каждый вызов stub.

## 4. Как организован деплой без простоя (Zero Downtime)?

**В Docker Compose** — zero downtime не реализован. Это инструмент для локальной разработки, не для продакшн-обновлений.

**В Kubernetes** — используется стратегия `RollingUpdate` (дефолтная для Deployment):
1. K8s поднимает новый под с обновлённым образом.
2. Ждёт, пока readiness probe вернёт успех.
3. Только после этого убивает старый под.
4. Трафик не прерывается — в любой момент жив хотя бы один под.

В CI/CD пайплайне после `kubectl apply` запускается `kubectl rollout status --timeout=120s` — он блокирует пайплайн до успешного завершения роллаута и откатывает (`kubectl rollout undo`) при неудаче.

**Ограничение текущей реализации:** `likes-svc` хранит данные in-memory. При рестарте пода данные теряются. Для настоящего zero downtime нужна внешняя БД (Redis/PostgreSQL), к которой подключается любой экземпляр сервиса.

## 5. Что было самым сложным в интеграции?

Самым нетривиальным оказалось **управление proto-стабами между сервисами**.

`likes_pb2.py` и `likes_pb2_grpc.py` генерируются из `.proto` и нужны в обоих сервисах — в `likes-svc/app/` и в `gateway/proto_stubs/`. При этом они должны быть синхронизированы: если изменить proto и перегенерировать только в одном месте, сервисы станут несовместимы.

Решение: команда `make proto` перегенерирует стабы и сразу копирует их в оба места. В CI это первый шаг перед сборкой Docker-образов.

Второй сложный момент — **версионирование grpcio**. Генератор `grpcio-tools` создаёт код под конкретную версию библиотеки. Если в `Dockerfile` установлена другая версия, контейнер падает с ошибкой версионной несовместимости. Решение: явно пинить версию `grpcio>=1.78.0` в `requirements.txt`.

## 6. Что бы вы улучшили, если бы был ещё месяц?

1. **Persistence** — заменить in-memory dict на PostgreSQL или Redis. Это главное ограничение текущей реализации: рестарт сервиса = потеря всех данных.

2. **Readiness/Liveness probes в K8s** — добавить `httpGet /health` для gateway и gRPC health protocol для likes-svc, чтобы Kubernetes не роутил трафик на не-готовые поды.

3. **Observability** — добавить Prometheus-метрики (количество лайков, latency gRPC-вызовов) и structured logging в JSON для сбора через Loki/ELK.

4. **Helm chart** — параметризовать K8s-манифесты через Helm, чтобы одной командой деплоить в dev/staging/prod с разными значениями (replicas, image tag, resource limits).

5. **Retry + timeout на gRPC** — сейчас gateway вызывает likes-svc без таймаута и без retry. В реальном сервисе нужен `timeout=2.0` и 2-3 попытки с exponential backoff.
